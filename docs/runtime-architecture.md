# Runtime architecture and cleanup

This note records the runtime foundation. See also [`vision.md`](vision.md) for
the higher goal: `graft` as direnv for containers with an enter/leave/idle
lifecycle.

## Current working slice

The first CLI slice conceptually does:

```text
CLI
  -> temporary rootfs directory
  -> temporary Quadlet .container in $XDG_RUNTIME_DIR/containers/systemd
  -> systemctl --user daemon-reload/start
  -> Podman runs with Rootfs=
  -> show logs
  -> clean up temporary files
```

No `Image=` is used. The container uses:

```text
Rootfs=<temp-rootfs>
Volume=/nix/store:/nix/store:ro
Exec=<absolute /nix/store/... command>
```

This only proves that the runtime chain works:

```text
Quadlet -> systemd --user -> Podman -> rootfs-store command
```

## Desired flow

The flow is:

```text
TOML
  -> load
  -> normalize
  -> resolve graph
  -> validate
  -> effective config
  -> render Quadlet
  -> apply/run via systemd
  -> cleanup according to lifecycle policy
```

## Separation of concerns

The implementation should grow two core modules:

```text
config engine
  load TOML, normalize, resolve graph, merge, validate

runtime manager
  write Quadlet files, systemd reload/start/stop, show logs, cleanup
```

`run-rootfs` is temporarily one consumer of the runtime manager. Later
`run <entry>` should call the same runtime manager based on effective TOML
config.

## Minimal TOML run

The first TOML-driven step can still be without a graph:

```toml
version = 1
name = "hello"

[config.runtime]
mode = "rootfs-store"
packages = ["bashInteractive", "coreutils"]
command = ["bash", "-lc", "echo hello from toml"]
```

Available commands:

```bash
graft render ./examples/rootfs-store.toml
graft run ./examples/rootfs-store.toml
graft up ./examples/rootfs-store.toml
```

Only then add graph resolving and project sessions:

```bash
graft up
graft enter
graft leave
```

## Lifecycle: transient versus managed

There are two main modes.

### Transient run

For agent runs, smoke tests, and one-shot jobs.

```text
Quadlet location: $XDG_RUNTIME_DIR/containers/systemd
rootfs/tmp:        $XDG_RUNTIME_DIR/graft/runs/<run-id>
cleanup:           by default after exit
```

### Managed service

For real long-running containers.

```text
Quadlet location: ~/.config/containers/systemd
state:            persistent unless explicitly removed
cleanup:          only via apply/remove/cleanup command
```

System-level management under `/etc/containers/systemd` can be added later, but
the first implementation stays rootless user-mode.

## Cleanup objects

The tool must reason explicitly about what is temporary or persistent.

### Temporary Quadlet files

```text
$XDG_RUNTIME_DIR/containers/systemd/*.container
```

Transient runs clean these up afterwards.

### Temporary rootfs/run directories

```text
$XDG_RUNTIME_DIR/graft/runs/<run-id>/
```

Cleaned up after exit by default. A debug/keep mode can be added later.

### Podman containers

Quadlet/systemd normally runs `podman rm` on stop when `--rm` is used. Even so,
the tool must be able to detect stale containers.

### Volumes and state

Volumes must not be discarded implicitly. A per-volume lifecycle should be
possible later:

```toml
volumes:
  - name: agent-home
    lifecycle: transient
  - name: cache
    lifecycle: persistent
```

### Logs

Journald logs are normally retained. The tool shows logs via:

```bash
journalctl --user -u <unit>
```

Logs are not actively deleted by the first cleanup flow.

## Labels and naming

All generated resources must be recognisable as owned by this tool.

Naming convention:

```text
graft-<entry-or-kind>-<run-id>
```

Podman labels:

```ini
Label=io.github.zerodawn.graft=true
Label=io.github.zerodawn.graft.entry=<entry>
Label=io.github.zerodawn.graft.run-id=<run-id>
Label=io.github.zerodawn.graft.lifecycle=transient|persistent
```

Quadlet header:

```ini
# Generated by graft
# entry=<entry>
# run-id=<run-id>
# lifecycle=transient|persistent
```

Cleanup may only remove resources marked this way.

## Desired commands

First:

```bash
graft render <toml-or-entry>
graft run <toml-or-entry>
graft cleanup
```

Later:

```bash
graft cleanup --older-than 24h
graft cleanup --all-transient
graft ps
graft status <entry>
graft apply <entry>
graft remove <entry>
```

## Implementation order

1. Keep the existing rootfs/Quadlet slice as the runtime foundation.
2. Add TOML loading for a single file, still without a graph.
3. Implement `render <file>` to Quadlet text.
4. Implement `run <file>` via the same runtime manager.
5. Add labels, headers, and lifecycle metadata.
6. Add `cleanup` for stale transient resources.
7. Only then add TOML graph resolving.
